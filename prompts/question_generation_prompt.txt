## Semantic Preservation Rules
- Preserve the exact SQL semantics. Do not broaden or narrow the meaning.
- Preserve table/entity roles, containment direction, ranking direction, grouping logic, aggregate semantics, filters, and limit semantics.
- Preserve every explicit distance threshold, count threshold, date filter, comparison operator, and top-k value.
- Preserve the output shape implied by the SQL.
- If the SQL projects one scalar aggregate, ask for one value, not a list of details.
- If the SQL projects one non-aggregate field, ask for that field only; do not imply extra attributes or full records.
- If the SQL projects two or three fields, mention the output attributes naturally, especially for grouped outputs such as "for each X, how many Y".
- If the SQL uses DISTINCT, preserve the deduplication intent with wording such as distinct, unique, or equivalents when natural.
- If the SQL includes ORDER BY or LIMIT, make the ranking/top-k intent explicit in the question. Do not hide a top-k query behind a generic “list all” request.
- If the SQL is limited to one row because of ranking, nearest, or top-k logic, prefer singular/top-1 phrasing over plural listing language.
- If the SQL returns geometry, shape, boundary, centroid, or coordinates, mention that output explicitly. Otherwise do not ask for geometric output.
- Rephrase spatial functions into natural language. Do not expose raw SQL or PostGIS function names.
- Translate coordinate or extent accessors into natural geographic wording when possible:
  - `ST_XMin` / `ST_XMax`: west/east extent, longitude threshold, westernmost/easternmost
  - `ST_YMin` / `ST_YMax`: south/north extent, latitude threshold, southernmost/northernmost
- Prefer short spatial wording such as within, inside, intersecting, crossing, touching, nearest to, or closest to.
- Only describe relations supported by the SQL. Do not invent entities, columns, filters, joins, or assumptions.
- Keep geometry/geography wording consistent with the actual SQL meaning.
- Use benchmark-like surface forms. Common valid forms include:
  - a short direct question: "What ...", "Which ...", "How many ...", "List ...", "Find ...", "Return ...", or "Show ..."
  - a direct question plus a short parenthetical disambiguation such as `(identified by GEOID starting with 12)`
  - a direct question plus one short clarification sentence when the SQL depends on a definition, alias, segmented named feature, or output restriction
- Add a short clarification only when it is semantically necessary, for example to express:
  - identifier or prefix filters such as `identified by`, `STATEFP = ...`, or `GEOID starting with ...`
  - definitions such as `refers to ...`
  - segmented named features such as `is composed of multiple sections with the same name`
  - output restrictions such as `Only provide the area` or `Return their GEOIDs`
- Do not mention SQL, PostGIS, databases, schemas, field names, aliases, CTE names, or implementation details.
- Generate exactly one question, not a question set.
- Do not show analysis, drafts, bullet points, self-checks, or alternative phrasings.
- Keep the question concise and natural. Prefer one sentence, but a short second sentence is allowed when it preserves meaning more faithfully. Avoid long stacked explanations.
- Use first-person or mild polite wording only if the requested style calls for it.
